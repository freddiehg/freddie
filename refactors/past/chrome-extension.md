# the mercury Chrome extension

`chrome-extension/` pushes Chrome's active tab URL to a running mercury, which is how the site layer knows what site you are on (`chrome-tab-url.md`). It sends one message and takes none. mercury's side of that wire is `external-events.md`; commands going the other way are still `external-effects.md`.

TypeScript under pnpm, held to the posture the Rust is: `strict` plus the flags past it, and typescript-eslint's `strictTypeChecked` rather than `recommended`, because the workspace denies clippy's `pedantic` and `nursery` and the browser half should not be the soft spot.

## The wire type is mercury's

`src/wire/` is generated by `ts-rs` from the same `IncomingEvent` mercury parses frames with, so `background.ts` imports the type rather than restating it, and a variant renamed in Rust fails `tsc`.

```ts
export type IncomingEvent = { "kind": "IncomingEvent.Tab", "value": TabMessage };
export type TabMessage = { url: string, };
```

That is the project's discriminated-union form, produced by `serde-compat` reading the `serde` attributes rather than by ts-rs's own defaults. Rust doc comments come through into the `.ts` as well.

The derive is behind mercury's `typescript` feature, off in every normal build, and the export runs during a test:

```
cargo test -p mercury --features typescript export_bindings
```

`export_to` is `../../../chrome-extension/src/wire/`, three levels, because ts-rs resolves it relative to `<crate>/bindings/` rather than the crate root. Two levels writes to `crates/chrome-extension/`, which is what the first attempt did.

The output is checked in, so loading the extension needs no Rust toolchain, and both places that already run the tests keep it honest without a new command: `.pre-commit-config.yaml`'s `cargo-test` hook and CI's `cargo-test` job both run `cargo test --all --all-features`, and `--all-features` turns the feature on. Pre-commit needs nothing configured, because it refuses a commit whose hooks changed a file. CI has one added step after the tests:

```yaml
      - name: Wire types are up to date
        run: git diff --exit-code chrome-extension/src/wire/
```

## What is where

```
chrome-extension/
  README.md            build, install, configure, and how to tell it is working
  manifest.json
  options.html
  package.json         pnpm, dev dependencies only
  pnpm-lock.yaml
  tsconfig.json
  eslint.config.js
  .prettierignore
  src/background.ts    listeners, socket, and the frame it sends
  src/options.ts       the port setting
  src/wire/            generated, checked in, never edited
  dist/                tsc output, gitignored, what the manifest points at
  icons/               copied from the menu bar's asset by the build, gitignored
```

Chrome loads `chrome-extension/` and the manifest points into `dist/`, so `pnpm build` has to have run before it will load at all. `pnpm watch` covers the edit loop, and rebuilding does not reload the extension: that is the reload arrow on its card.

The icon is the menu bar's, copied out of `crates/mercury/assets/` by `pnpm run icons`, which `build` and `watch` both depend on. A second copy in git would be a thing to keep in step by hand, and a build was already mandatory.

## The pushes

Three listeners, because tab events alone do not cover which tab is front:

- `chrome.tabs.onActivated` for a tab switch.
- `chrome.tabs.onUpdated`, filtered to `tab.active` and to changes that carried a `url`.
- `chrome.windows.onFocusChanged`, ignoring `WINDOW_ID_NONE`. Returning from another application, and switching between two Chrome windows, both change which tab is front with no tab event at all.

`connect` returns the socket it opened rather than leaving the caller to re-read the module variable, which could by then hold a different one. The `close` and `error` handlers clear only their own socket: a connection that fails fires both, and by then a later tab event may already have replaced it, so clearing unconditionally would drop a live socket and strand its pending send.

A URL that cannot be sent is dropped rather than queued. The next tab event supersedes it, so a retry would deliver a stale answer to a question nobody asked yet. Re-sending an identical URL is harmless: `record_tab_url` writes the same value and dispatch produces no change.

The port is read from `chrome.storage.local` on every push rather than cached, so changing it on the options page takes effect on the next tab event with nothing to reload.

## What strict TypeScript caught

Both on the first build, both real:

- `const status = document.querySelector(...)` in `options.ts` collides with `window.status`, which is a global `string`. Not shadowing, a redeclaration error, plus two more about `.textContent` not existing on `string`. It is `statusLine` now.
- Top-level `await` needs the file to be a module, so `options.ts` carries an `export {}` that exists for that and says so.

`@types/chrome` had to be `0.2.2`, not the `0.0.279` the plan named. The 0.0.x line marks the entire `chrome` namespace `@deprecated` over the old Chrome Apps platform, so `strictTypeChecked`'s `no-deprecated` rule failed on every API call in the extension.

## Verified

Loaded unpacked, with mercury running. `~/Library/Logs/mercury/mercury.log` records a dispatch per tab switch whose state carries the real URL, arriving from the extension rather than from a test client.

`pnpm build`, `pnpm typecheck`, `pnpm lint`, and `pnpm format:check` all pass, and CI runs the last three as an `extension` job that `all-checks-passed` depends on, so a broken extension blocks a release the way a broken crate does.

## Still open

A single page load produces several identical frames, five for one `x.com/home` load in the log. Harmless, since an identical URL is a no-op through dispatch, and worth deduplicating against the last URL sent if it ever gets noisy.

`onUpdated` covers document loads. It is not the event for a same-document navigation, which is what an SPA does through the History API or a fragment; `chrome.webNavigation.onHistoryStateUpdated` and `onReferenceFragmentUpdated` exist for those. This costs nothing while `Site::from_url` matches on the host, since a route change inside `claude.ai` leaves the host alone, and it starts mattering the moment a bind keys on the path.

Whether `onFocusChanged` fires when focus returns from another application, rather than only between Chrome windows, and whether devtools taking focus produces a spurious push.

## Other browsers

One extension serves the Chromium browsers that support MV3 with the same `chrome.*` APIs (Chrome, Brave, Arc). They differ only in bundle id, which is mercury's concern through `App::from_bundle_id`, not the extension's. Safari has no equivalent loopback-WebSocket path and is out of scope.
