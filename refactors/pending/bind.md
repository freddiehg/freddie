# bind: the #[bind] derive

This doc is the implementation detail of the binding layer: the `#[binds]`/`#[bind]` attributes, the `EventHandler<M>` impls the derive generates, and how accumulation and dispatch run over a laserbeam tree. The outer loop (the merged event stream, diffing the registered set against the OS, the registrar) is in `freddie-keys-plan.md`. This doc assumes no clobbering: a trigger is bound at most once on any active path.

bind is a crate within freddie: `bind` (the traits and the walks) plus `bind_macro` (the `#[derive(Bind)]`). It builds on laserbeam: bindability follows laserbeam's descent edges (see "What is bindable").

## The marker and the two traits

The app implements `Bindings` on one marker type, naming the unified `Trigger` enum and the event and output types every handler uses:

```rust
trait Bindings {
    type Trigger: Hash + Eq + Clone;   // enum Trigger { Keyboard(Keyboard), Foreground(Foreground), .. }
    type Event;
    type Output;
}

struct MercuryStruct;
impl Bindings for MercuryStruct {
    type Trigger = Trigger;
    type Event   = MercuryEvent;
    type Output  = Option<Vec<MercuryEffect>>;
}
```

Every bindable node implements `EventHandler<M>`, generated by `#[derive(Bind)]`:

```rust
trait EventHandler<M: Bindings> {
    // add this node's triggers, and those of its active descendants, to `out`.
    fn accumulate(&self, out: &mut HashSet<M::Trigger>);
    // dispatch is generated as an inherent fn per node, not a trait method; see "Dispatch".
}
```

## The attributes

```rust
#[derive(Laserbeam, Bind)]
#[binds(MercuryStruct)]
#[bind(Keyboard::new('g'), on_g)]
#[bind(Keyboard::new('y'), on_y)]
struct Inner {
    #[resolve_into] child: Child,   // the one bindable sub-node
    label: String,                  // plain state, not bindable
}
```

- `#[binds(MercuryStruct)]` names the marker `M`. The derive emits `impl EventHandler<MercuryStruct> for Inner`. Because `M` is concrete, `M::Trigger`/`Event`/`Output` are concrete in the impl, so `.into()` lifts each trigger into `Trigger` and the handler signatures `fn(MercuryEvent, Path<Inner, _>) -> Option<Vec<MercuryEffect>>` type-check.
- `#[bind(trigger, handler)]`, repeatable. `trigger` lifts into `M::Trigger` via `Into`; `handler` is `fn(M::Event, Path<Self, Parent>) -> M::Output`.

## What is bindable

The question "is `EventHandler` implemented for everything, even a `String` field?" — no. Bindability follows laserbeam's descent edges, nothing else:

- For a struct, the only bindable children are its `#[resolve_into]` fields. Plain fields (`String`, counters, buffers) are state; they are never recursed into and never need `EventHandler`.
- For an enum, the bindable child is the active variant's inner type. Every variant's inner type is assumed to implement `EventHandler<M>` (it derives `Bind`).

So `EventHandler<M>` is implemented only on the node types the user derives `Bind` on, and the walks only ever follow `#[resolve_into]` fields and variant inners. That edge set is exactly laserbeam's resolve graph, which is why bind builds on laserbeam: the edges to recurse are the ones laserbeam already computes.

## Accumulation

`accumulate` produces the `HashSet<M::Trigger>` to register for the current state, walking the active path's descent edges. The derive generates it per node:

```rust
// struct with its own binds and one #[resolve_into] child:
impl EventHandler<MercuryStruct> for Inner {
    fn accumulate(&self, out: &mut HashSet<Trigger>) {
        for t in [Keyboard::new('g').into(), Keyboard::new('y').into()] {
            if !out.insert(t) { /* v1: same trigger bound twice on the active path -> error */ }
        }
        self.child.accumulate(out);   // recurse the #[resolve_into] edge; self.label is ignored
    }
}

// enum: own (enum-level) binds, then the active variant's inner:
impl EventHandler<MercuryStruct> for Layer {
    fn accumulate(&self, out: &mut HashSet<Trigger>) {
        // ... any enum-level #[bind]s ...
        match self {
            Layer::Nav(n)    => n.accumulate(out),
            Layer::Typing(t) => t.accumulate(out),
        }
    }
}
```

- No clobbering: `HashSet::insert` returning `false` means the trigger is already bound elsewhere on the active path, which is an error, raised here.

Entry point on the root: `root.accumulate::<MercuryStruct>()` allocates the set, calls the trait method, and returns the `HashSet<Trigger>`. The runtime diffs it against the previous state's set and hands the delta to the registrar (keys-plan).

## Dispatch

Accumulation is for registration (keys only). Dispatching a fired event is a separate walk that runs the handler with a live path, so there is no stored handler map and no type erasure.

On a fired `(Trigger, Event)`:

1. `<Root as Resolve>::resolve(&mut root)` gives the active leaf path (laserbeam's `Resolved`).
2. The root's generated entry matches `Resolved` to the concrete leaf path and calls `Leaf::dispatch_from(leaf_path, &fired, event)`.
3. `dispatch_from` checks this node's binds; if one matches `fired` it calls that handler with the path (consuming it) and returns. Otherwise it ascends via `into_parent` and calls the parent's `dispatch_from`. With no clobbering, at most one node on the active path binds `fired`, so the first match is the only one; if none match, the chain reaches the root and returns `None`.

```rust
// generated per node, inherent (the path type is node-specific):
impl Inner {
    fn dispatch_from(path: Path<Inner, ParentPath>, fired: &Trigger, ev: MercuryEvent)
        -> Option<Option<Vec<MercuryEffect>>>
    {
        if *fired == Keyboard::new('g').into() { return Some(on_g(ev, path)); }
        if *fired == Keyboard::new('y').into() { return Some(on_y(ev, path)); }
        ParentNode::dispatch_from(path.into_parent(), fired, ev)   // not bound here -> ascend
    }
}
```

The binding is only compared (a `Trigger` equality check) before the path is consumed, so an unmatched node still owns its path to ascend; only the matching node consumes it into its handler. The root's `dispatch_from` has no parent, ending the chain (returns `None` if nothing matched).

## Wrinkles and open points

- `dispatch_from` cannot be a uniform `EventHandler<M>` method, because each node's path type differs. Either a GAT `type Path<'a>` on the trait, or keep `dispatch_from` inherent and let the root entry tie the chain together. Leaning inherent plus a generated root entry.
- `Event` threads by value down the ascend chain and moves into the matching handler; `fired` passes by reference, compared at each level.
- The accumulate match on the active enum variant duplicates laserbeam's descent. Decide whether the derive re-emits the match or calls into laserbeam to get the active edge.
- The recursion edges are laserbeam's `#[resolve_into]` fields and variant inners.
- The loop, the registrar, and the diffing live in `freddie-keys-plan.md`; this doc stops at `accumulate` (the set for a state) and `dispatch` (one fired event).
